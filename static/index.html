Algorithms
How each algorithm works under the hood
sliding_windowZSET · O(log N)

Stores each request as a member of a Redis sorted set, scored by its timestamp in milliseconds. On every request: (1) atomically remove all members with score < now - window_ms, (2) count remaining, (3) if below limit, add new member and allow.

ZREMRANGEBYSCORE key -∞ (now - window)
ZCARD key → count
if count < limit: ZADD key now member
-- all in one atomic Lua script
no boundary burst O(log N) per request memory ∝ request volume
fixed_windowSTRING · O(1)

Counts requests in fixed time buckets. The key is rl:fw:{client}:{floor(now/window)}. Increment with INCR, set TTL on first write. Fastest possible — one Redis round-trip.

bucket = floor(now_ms / window_ms)
INCR rl:fw:{client}:{bucket}
if count == 1: EXPIRE key window+1
if count <= limit: allow
lowest latency O(1) memory 2× burst at boundary
token_bucketHASH · O(1)

Stores {tokens, last_refill} in a Redis Hash. On every request, virtually refills tokens based on elapsed time, then consumes one. No background worker — refill is computed on read.

elapsed = (now - last_refill) / 1000
tokens = min(capacity, tokens + elapsed × rate)
if tokens ≥ 1: tokens -= 1; allow
HMSET key tokens {tokens} last_refill {now}
burst tolerance O(1) memory per client smooth refill
