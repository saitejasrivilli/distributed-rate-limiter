Algorithms
How each algorithm works at the Redis level
sliding_window
ZSETO(log N)no boundary burst

Each request is stored as a scored member in a Redis sorted set, where the score is the current timestamp in milliseconds. On every request, a Lua script atomically removes all entries older than the window, counts the remaining entries, and if under the limit, adds the new request.

-- atomic Lua script (single round-trip)
ZREMRANGEBYSCORE key -inf (now - window_ms)
count = ZCARD key
if count < limit:
  ZADD key now "{now}-{seq}" -- seq avoids collision
  PEXPIRE key window_ms + 1000
  return {1, limit - count - 1, 0}
else:
  retry = oldest_score + window_ms - now
  return {0, 0, retry}

fixed_window
STRINGO(1)lowest latency

Requests are counted in fixed time buckets. The Redis key includes the bucket number derived from the current timestamp, so keys auto-rotate without any cleanup. One INCR per request — the fastest possible implementation.

bucket = floor(now_ms / window_ms)
key = "rl:fw:{client}:{bucket}"
count = INCR key
if count == 1: EXPIRE key window_sec + 1
if count <= limit: allow
-- tradeoff: 2× burst possible at window boundary

token_bucket
HASHO(1)burst-tolerant

Stores tokens and last_refill in a Redis Hash. On every request, the script virtually refills based on elapsed time (no background worker needed), then consumes one token if available.

data = HMGET key "tokens" "last_refill"
elapsed = (now - last_refill) / 1000.0
tokens = min(capacity, tokens + elapsed × rate)
if tokens >= 1:
  tokens -= 1
  HMSET key tokens {tokens} last_refill {now}
  return allow
-- O(1) memory regardless of request volume
